# 20260415 적립(EARN) 로직 수정 내역

## 수정 배경

`kafka.md` 설계 원칙에 따라 적립 처리 방식을 **동기 처리 → Kafka 비동기 처리**로 전환한다.

기존 구현은 API 서버에서 Redis 락 + DB 저장까지 모두 동기로 처리한 뒤 Kafka에 사후 이벤트를 발행하는 구조였다.
이를 kafka.md 설계 의도에 맞게 API 서버는 Kafka 발행만 담당하고, 실제 적립 처리는 Consumer가 비동기로 수행하도록 수정했다.

---

## 변경 전 흐름 (AS-IS)

```
클라이언트 요청
  └─ PointController
       └─ PointEarnUseCase
            ├─ Redis 락 획득 (5초 TTL)
            ├─ requestId 중복 체크 (DB 조회)
            ├─ JPA PESSIMISTIC_WRITE 로 지갑 잠금
            ├─ PointWallet / PointLedger 저장 (동기)
            └─ Kafka point-earned 발행 (사후 알림용)
  └─ 200 OK + 잔액 즉시 반환
```

---

## 변경 후 흐름 (TO-BE)

```
클라이언트 요청
  └─ PointController
       └─ PointEarnUseCase
            └─ Kafka point-earn 발행 (key = memberId)
  └─ 202 Accepted + requestId 반환

         [비동기]
  └─ PointEarnKafkaConsumer  ← point-earn 토픽 수신
       └─ EarnPointProcessor
            ├─ requestId 중복 체크 (DB 조회)
            ├─ 중복이면 무시
            └─ PointWallet / PointLedger 저장 (트랜잭션)
```

## 수정 파일 목록

### 1. `PointEarnUseCase.java` — 수정

**경로:** `src/main/java/com/study/point/application/point/PointEarnUseCase.java`

| 항목 | 변경 내용 |
|------|-----------|
| Redis 락 | **제거** — 적립에서는 사용하지 않음 (kafka.md 설계 원칙) |
| requestId 중복 체크 | **제거** — Consumer(EarnPointProcessor)로 이동 |
| DB 저장 로직 | **제거** — Consumer(EarnPointProcessor)로 이동 |
| Kafka 발행 | **유지** — `sendEarnCommand()` 호출로 변경 |
| `@Transactional` | **제거** — DB 작업 없음 |
| 반환 타입 | `PointResponse` → `void` |

```java
// 변경 후
@Service
@RequiredArgsConstructor
public class PointEarnUseCase {

    private final PointEventProducer pointEventProducer;

    public void earn(EarnPointCommand command) {
        pointEventProducer.sendEarnCommand(command);
    }
}
```

### 2. `PointEventProducer.java` — 수정

**경로:** `src/main/java/com/study/point/infrastructure/kafka/PointEventProducer.java`

| 항목 | 변경 내용 |
|------|-----------|
| 기존 `publishEarned(PointLedger)` | **제거** — 사후 알림 발행 불필요 |
| 토픽 | `point-earned` → `point-earn` |
| 발행 key | `walletId` → `memberId` (순차처리 기준 통일) |
| 발행 value | `PointLedger` 엔티티 → `EarnPointCommand` |
| 신규 메서드 | `sendEarnCommand(EarnPointCommand)` 추가 |

```java
// 변경 후
private static final String TOPIC_EARN = "point-earn";

public void sendEarnCommand(EarnPointCommand command) {
    kafkaTemplate.send(TOPIC_EARN, String.valueOf(command.memberId()), command);
}
```

> **Kafka key = memberId** 이유: 같은 회원 요청이 동일 파티션으로 라우팅되어 Consumer에서 순차 처리된다.

### 3. `EarnPointProcessor.java` — **신규 생성**

**경로:** `src/main/java/com/study/point/application/point/EarnPointProcessor.java`

Consumer에서 위임받아 실제 적립을 처리하는 서비스 계층.

| 처리 순서 | 내용 |
|-----------|------|
| 1 | requestId로 PointLedger 조회 → 이미 존재하면 무시(멱등 처리) |
| 2 | PointPolicyPort로 적립 정책 로드 |
| 3 | memberId로 PointWallet 조회 (없으면 신규 생성) |
| 4 | 정책 검증 후 wallet.earn() 호출 |
| 5 | PointLedger 생성 및 저장 |
| 6 | PointWallet 저장 |

```java
@Slf4j
@Service
@RequiredArgsConstructor
@Transactional
public class EarnPointProcessor {
    // ...
    public void process(EarnPointCommand command) {
        if (pointLedgerRepository.findByRequestId(command.requestId()).isPresent()) {
            log.info("Duplicate earn request ignored: requestId={}", command.requestId());
            return;
        }
        // ... PointWallet / PointLedger 저장
    }
}
```

### 4. `PointEarnKafkaConsumer.java` — **신규 생성**

**경로:** `src/main/java/com/study/point/infrastructure/kafka/PointEarnKafkaConsumer.java`

| 항목 | 내용 |
|------|------|
| 구독 토픽 | `point-earn` |

```java
public void consume(EarnPointCommand command) {
    log.info("Received earn command: requestId={}, memberId={}", command.requestId(), command.memberId());
    earnPointProcessor.process(command);
}
```

### 5. `PointController.java` — 수정

**경로:** `src/main/java/com/study/point/api/point/PointController.java`

| 항목 | 변경 내용 |
|------|-----------|
| HTTP 상태 코드 | `200 OK` → `202 Accepted` |
| 응답 바디 | `PointResponse`(잔액 포함) → `{ "requestId": "..." }` |
| 이유 | 비동기 처리이므로 즉시 잔액을 알 수 없음 |

```java
// 변경 후
@PostMapping("/earn")
public ResponseEntity<ApiResponse<Map<String, String>>> earn(@Valid @RequestBody EarnPointRequest request) {
    String requestId = request.requestId() != null ? request.requestId() : UUID.randomUUID().toString();
    EarnPointCommand command = new EarnPointCommand( /* ... */ requestId );
    pointEarnUseCase.earn(command);
    return ResponseEntity.status(HttpStatus.ACCEPTED)
            .body(ApiResponse.ok(Map.of("requestId", requestId)));
}
```

## 역할 분리 정리

| 계층 | 클래스 | 역할 |
|------|--------|------|
| API | `PointController` | 요청 검증, 202 Accepted 반환 |
| Application | `PointEarnUseCase` | Kafka 발행 위임 |
| Infrastructure | `PointEventProducer` | Kafka point-earn 토픽 발행 |
| Infrastructure | `PointEarnKafkaConsumer` | 메시지 수신 + 위임 |
| Application | `EarnPointProcessor` | requestId 중복 체크 + DB 저장 |

## kafka.md 설계 원칙 준수 여부

| 원칙 | 준수 여부 |
|------|-----------|
| 적립 요청은 Kafka를 통해 비동기 순차 처리 | ✅ |
| Kafka key는 memberId 또는 walletId | ✅ memberId 사용 |
| requestId는 Kafka key가 아닌 중복제어용 | ✅ |
| 적립 중복제어는 DB unique 제약으로 최종 보장 | ✅ PointLedger.requestId UNIQUE |
| Consumer는 수신과 위임 역할 중심 | ✅ |
| 도메인 로직은 UseCase/Service 계층에 | ✅ EarnPointProcessor |
| 적립에서 Redis 락 미사용 | ✅ 제거 |
