## 현재 코드에서 EarnPolicy의 역할

`EarnPolicy`는 `domain.point.vo` 패키지에 있는 작은 정책 VO(Value Object)다.

```java
public record EarnPolicy(long maxEarnPerOnce, long maxHoldFreePoint)
```

이 객체는 DB에 직접 매핑되는 JPA 엔티티가 아니다. `EarnPointProcessor`가 애플리케이션 계층의 `PointPolicyConfig`를 읽은 뒤, 도메인 계층에서 사용할 `EarnPolicy`로 변환하고, `PointWallet.earn(long amount, EarnPolicy policy)`에 전달한다.

흐름은 다음과 같다.

1. `EarnPointProcessor`가 `pointPolicyPort.loadPolicy()`로 정책 설정을 조회한다.
2. 조회한 `PointPolicyConfig`를 `EarnPolicy.of(...)`로 도메인 VO로 변환한다.
3. `PointWallet.earn(...)`이 `EarnPolicy`를 기준으로 1회 적립 한도와 지갑 보유 한도를 검증한다.

즉 `EarnPolicy`는 “적립 정책값을 도메인 로직에 전달하기 위한 불변 정책 객체” 역할이다.

## record를 사용한 이유

`EarnPolicy`는 `record`로 정의하는 것이 현재 코드 맥락에 잘 맞는다.

첫째, `EarnPolicy`는 식별자를 가진 엔티티가 아니라 값 객체다. JPA 엔티티처럼 `id`, 영속성 생명주기, 지연 로딩, 변경 메서드가 필요하지 않다.

둘째, 정책값은 생성 후 변경되지 않는 편이 안전하다. `record`의 컴포넌트는 기본적으로 `private final`이며, 생성 이후 값을 바꿀 수 없다. 적립 처리 중 정책값이 변경되는 구조보다 불변 객체로 전달하는 구조가 도메인 로직을 더 예측 가능하게 만든다.

셋째, 보일러플레이트가 줄어든다. `record`는 생성자, accessor, `equals`, `hashCode`, `toString`을 자동으로 제공한다. `EarnPolicy`처럼 순수하게 값을 묶는 객체에서는 일반 class보다 의도가 명확하다.

## of를 사용한 이유와 판단

기존 코드는 다음과 같았다.

```java
public static EarnPolicy of(long maxEarnPerOnce, long maxHoldFreePoint) {
    return new EarnPolicy(maxEarnPerOnce, maxHoldFreePoint);
}
```

현재 구현만 보면 `of()`는 `new EarnPolicy(...)`를 감싸는 얇은 wrapper였다. 따라서 기능적으로는 필수는 아니었다.

하지만 정적 팩토리 메서드를 두는 것은 다음 장점이 있다.

- 생성 의도를 이름으로 드러낼 수 있다.
- 나중에 검증, 정규화, 기본값 보정, 캐싱 같은 생성 규칙을 한곳에 모을 수 있다.
- 호출부가 생성자에 직접 의존하는 느낌보다 도메인 언어에 가깝다.

`README.md`에는 “생성 정적 팩토리: create”라고 적혀 있고, `PointWallet.create(...)`도 이 규칙을 따른다. 다만 이 규칙을 모든 객체에 강제하기보다는 다음 기준으로 나누는 편이 더 낫다고 판단했다.

- 엔티티 생성: `create`
- VO, DTO, 단순 값 조합: `of`

`create`는 생명주기를 가진 엔티티를 새로 만든다는 느낌이 강하고, `of`는 여러 값을 조합해 하나의 값 객체를 만든다는 의미가 강하다. 따라서 `EarnPolicy`는 `of`를 유지하는 쪽이 더 표현력이 좋다.

## 발견한 개선점

기존 `EarnPolicy`는 정책값 자체의 유효성을 검증하지 않았다.

그 결과 다음과 같은 잘못된 정책값도 객체로 생성될 수 있었다.

- `maxEarnPerOnce <= 0`
- `maxHoldFreePoint <= 0`
- `maxEarnPerOnce > maxHoldFreePoint`

`PointWallet.earn()`은 실제 적립 요청의 amount를 검증하지만, 정책값 자체가 정상인지까지 책임지는 위치는 아니다. 정책값의 불변 조건은 `EarnPolicy` 내부에서 막는 편이 도메인 VO의 책임에 더 맞다.

## 수정 내용

### 1. EarnPolicy compact constructor 추가

`record`의 compact constructor를 사용해 생성 시점 검증을 추가했다.

```java
public record EarnPolicy(long maxEarnPerOnce, long maxHoldFreePoint) {
    public EarnPolicy {
        if (maxEarnPerOnce <= 0) {
            throw new IllegalArgumentException("maxEarnPerOnce must be positive");
        }
        if (maxHoldFreePoint <= 0) {
            throw new IllegalArgumentException("maxHoldFreePoint must be positive");
        }
        if (maxEarnPerOnce > maxHoldFreePoint) {
            throw new IllegalArgumentException("maxEarnPerOnce must not exceed maxHoldFreePoint");
        }
    }

    public static EarnPolicy of(long maxEarnPerOnce, long maxHoldFreePoint) {
        return new EarnPolicy(maxEarnPerOnce, maxHoldFreePoint);
    }
}
```

이제 `EarnPolicy.of(...)`뿐 아니라 `new EarnPolicy(...)`로 직접 생성해도 동일한 검증이 적용된다. `record`의 canonical constructor에 검증을 넣었기 때문에 생성 경로가 달라져도 불변 조건이 깨지지 않는다.

### 2. EarnPolicy 단위 테스트 추가

`src/test/java/com/study/point/domain/point/vo/EarnPolicyTest.java`를 추가했다.

검증한 내용은 다음과 같다.

- 유효한 정책값이면 생성된다.
- `maxEarnPerOnce`는 양수여야 한다.
- `maxHoldFreePoint`는 양수여야 한다.
- `maxEarnPerOnce`는 `maxHoldFreePoint`보다 클 수 없다.

### 3. 컴파일 검증 중 발견한 memberId 타입 불일치 수정

`EarnPolicyTest`를 실행하기 위해 Gradle 테스트를 돌리던 중, 기존 코드의 컴파일 오류가 먼저 발견되었다.

```text
PointController.java: Long cannot be converted to String
```

원인은 `UsePointRequest.memberId`가 `Long`인데, `PointUseUseCase.use(...)`는 `String memberId`를 받는 구조였기 때문이다.

현재 코드베이스에서는 다음 타입들이 모두 `String memberId`를 사용한다.

- `PointWallet.memberId`
- `EarnPointRequest.memberId`
- `EarnPointCommand.memberId`
- `PointResponse.memberId`
- `PointWalletRepository.findByMemberId(String memberId)`

따라서 `UsePointRequest.memberId`도 `String`으로 맞추는 것이 더 일관적이다. 이에 따라 `UsePointRequest`의 `memberId`를 `String`으로 변경하고, 필수 문자열 검증을 위해 `@NotBlank`를 적용했다.

또한 `PointControllerTest`는 현재 컨트롤러 구현과 맞지 않는 오래된 기대값을 갖고 있었다.

- 적립 API는 현재 `200 OK + PointResponse`가 아니라 `202 Accepted + requestId`를 반환한다.
- `EarnPointRequest`는 현재 `pointType`까지 받는다.
- `PointResponse.memberId`는 `String`이다.

테스트가 현재 구현을 검증하도록 위 내용을 함께 정리했다.

## 성능 이슈 검토

`record`와 `static factory of()` 자체의 성능 부담은 사실상 무시 가능하다.

이유는 다음과 같다.

- `of()`는 정적 메서드 호출 후 생성자를 호출하는 단순 구조다.
- JVM JIT는 이런 작은 메서드를 쉽게 인라인할 수 있다.
- `EarnPolicy`는 `long` 필드 2개만 가진 작은 객체다.
- 현재 서비스의 주요 비용은 DB I/O, Kafka 처리, 트랜잭션, 네트워크 왕복이다.

따라서 이 프로젝트에서 병목은 `EarnPolicy` 객체 생성이 아니라 영속성/메시징/트랜잭션 경계에 있을 가능성이 압도적으로 높다.

다만 초고빈도 순수 메모리 연산 루프에서 수천만 번 이상 생성한다면 객체 할당 자체를 볼 수는 있다. 현재 포인트 적립 처리 흐름은 그런 성격이 아니므로 성능 때문에 `record`나 `of()`를 피할 이유는 없다.

## 최종 판단

가장 나은 방향은 다음과 같다.

1. `EarnPolicy`는 `record`로 유지한다.
2. `of()`도 유지한다. 단, 단순 wrapper로만 두지 않고 생성 규칙의 진입점으로 사용한다.
3. 정책값의 불변 조건은 `EarnPolicy` 내부에서 검증한다.
4. README의 naming convention은 추후 “엔티티는 create, VO는 of”처럼 더 구체화하는 것이 좋다.

이번 수정은 위 판단에 따라 `EarnPolicy`를 더 도메인 VO답게 만들고, 잘못된 정책 설정이 도메인 로직 안쪽으로 흘러 들어가지 않도록 막는 방향으로 진행했다.
